# Question 16

## Question
Your company has three virtual machines (VMs) that are included in an availability set.

You try to resize one of the VMs, which returns an allocation failure message.

It is imperative that the VM is resized.

Which of the following actions should you take?

- A. You should only stop one of the VMs.
- B. You should stop two of the VMs.
- C. You should stop all three VMs.
- D. You should remove the necessary VM from the availability set.

---

## Correct Answer
**C. You should stop all three VMs.**

---

## Explanation

When VMs are part of an **Availability Set**, Azure attempts to keep them grouped within the same cluster and hardware infrastructure.

An **allocation failure** during resizing usually occurs because:
- The current hardware cluster does not have enough capacity for the new VM size.

To successfully resize the VM:
1. Stop (deallocate) **all VMs in the availability set**
2. Azure then reallocates the availability set to a new cluster with sufficient capacity
3. Resize the required VM
4. Restart the VMs

This is the recommended Microsoft procedure for resizing VMs inside an availability set when allocation failures occur.

---

## Why the Other Options Are Incorrect

### A. You should only stop one of the VMs.
Incorrect.

Stopping only one VM does not free or relocate the entire availability set to another cluster.

---

### B. You should stop two of the VMs.
Incorrect.

All VMs in the availability set must be deallocated to allow Azure to move the set to compatible hardware.

---

### D. You should remove the necessary VM from the availability set.
Incorrect.

Removing a VM from an availability set is not a simple operation and is not the recommended solution for allocation failures during resizing.

---

- C. `select * from Event where EventType == "error"`
- D. `search in (Event) * | where EventType -eq "error"`

---

## Correct Answer
**B. `search in (Event) "error"`**

---

## Explanation

Azure Log Analytics uses the **Kusto Query Language (KQL)**.

The query:

```kusto
search in (Event) "error"
```

searches the `Event` table for records containing the term `"error"`.

This is the correct syntax for searching log data in Azure Log Analytics.

---

## Why the Other Options Are Incorrect

### A. `Get-Event Event | where {$_.EventType == "error"}`
Incorrect.

This is PowerShell syntax, not Kusto Query Language (KQL).

---

### C. `select * from Event where EventType == "error"`
Incorrect.

This is SQL syntax, which is not supported in Azure Log Analytics queries.

---

### D. `search in (Event) * | where EventType -eq "error"`
Incorrect.

Although closer to KQL syntax, `-eq` is PowerShell syntax, not KQL syntax.

- C. The App Service plan for WebApp1 remains in West Europe. Policy1 applies to WebApp1.

**A. The App Service plan for WebApp1 remains in West Europe. Policy2 applies to WebApp1.**

When you move an Azure Web App (App Service) between resource groups:

### 1. App Service Plan Behavior
- The **App Service plan does NOT move regions**
- It remains in its original region (West Europe in this case)

Azure resources are tied to a region at creation, and App Service plans cannot be relocated by resource group move.

---

### 2. Resource Group Policy Behavior
- Azure Policy is applied at the **resource group scope**
- After moving WebApp1 from RG1 to RG2, it becomes subject to:
  - **Policy2 (RG2 policy)**

---

### Final Outcome:
- App Service Plan stays in **West Europe**
- WebApp1 now falls under **RG2 policies (Policy2)**

## Why the Other Options Are Incorrect

### B. App Service plan moves to North Europe
Incorrect.

App Service plans cannot change regions simply by moving resource groups.

---

### C. Policy1 applies to WebApp1

Once moved to RG2, RG2’s policy replaces RG1’s policy.
